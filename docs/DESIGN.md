## Problem Statement

AMCPの中核データ構造である **IEDIレコード（Intent / Evidence / Delta / Insight）** を、以下の3つの利用目的に対して評価・設計する。

1. **信用情報**：取引相手として信頼できるか（外部取引向け）
2. **相性判断**：協働・コラボレーションの適合性評価
3. **自動化・高品質化**：内部作業（コーディング・方針決定・バックオフィス）の改善

さらに、内外統一利用を前提としたフィールド設計の改良案、および最初に実装すべき最小スコープを導出する。

---

## What Makes This Cool

**Intent 事前宣言 + Delta 測定** の組み合わせは、既存のすべての信用評価・ログシステムにない設計である。「言ったことをやった」という客観証明は、事前のコミットメント（Intent）なしには原理的に成立しない。IEDI はこれをプロトコルレベルで強制する。

さらに **Delta がスコアではなく構造化差分** であることが、スカラー値圧縮（ERC-8004、クラウドソーシング評価）との本質的な差別化を生む。「どの側面で誠実か」という文脈情報が保持される。

---

## Constraints

- AMCPはMCPのProfile仕様（MCP互換性を破らない）
- IEDI構造はプロトコル仕様書（docs/amcp-whitepaper.md v0.1、2026-05-06）として先行技術開示済み
- 実装はRust / TypeScript / Python いずれでも可（MCP SDK依存）
- 双方向ハッシュ検証はサードパーティ不要で成立する設計を維持

---

## Premises（前提確認）

1. **Intent 事前宣言は load-bearing（取り外し不可）**  
   Delta = 「事前に言ったこと vs 実際に起きたこと」の差分。Intent を事後に書いた時点でDeltaは後付け評価になり、既存スコアシステムと本質的に変わらなくなる。

2. **内部・外部で同一スキーマを使う**  
   「社内コーディング作業のIEDI」と「外部取引のIEDI」が同形式であることで、個人・組織の「意図実現精度」が横断的に分析できる。別スキーマにするとこの比較が消える。

3. **相性判断（用途2）には現行IEDIで構造的な欠缺がある**  
   現行Insightは自由テキストのため、「コミュニケーション頻度」「エスカレーション閾値」「価値観の合致度」をクエリで取り出せない。追加フィールドが必要。

## Q1：IEDI構造の適切性 — 競合フレームワークとの比較分析

### 比較表

| フレームワーク | Intent事前宣言 | Delta測定 | 双方向Insight | 用途1信用 | 用途2相性 | 用途3自動化 |
|---|:---:|:---:|:---:|:---:|:---:|:---:|
| **TAOL** (Trust-Action-Outcome-Learning) | ❌ | ❌ | ❌ | 弱い | ❌ | ❌ |
| **SOAP** (医療記録型) | 部分的 | ❌ | ❌ | 弱い | ❌ | ❌ |
| **OKR / PDCA** | ✅ (Objective) | 部分的 | ❌ | ❌ (期次集計) | ❌ | ❌ |
| **IEDI** | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |

### IEDIの優位性

**vs TAOL：**  
TAOLは「Trust」を入力値として持つため、初回取引者への信用付与が循環論法になる（信用がないから信用できない）。IEDIはゼロから Delta 蓄積で信用を構築できる。また TAOLは Intent 事前宣言を持たないため Delta が計算できず、Outcome は事後評価にとどまる。（注：TAOLは本稿の比較分析用に構成した対照フレームワークであり、固有の既存仕様を指すものではない。）

**vs SOAP：**  
SOAP（Situation-Objective-Action-Plan）の "Plan" は事後に書かれる次回方針であり、IEDIのIntentとは時制が逆。Intent は「実行前の約束」であり、「実行後の反省」ではない。この差が信用の根拠になる。

**vs OKR/PDCA：**  
OKRは期次集計であり、個別取引の改ざん検知・信用検証には使えない。「今週のスプリント目標を達成した」と「この1件の取引で言ったことをやった」は要求する精度が異なる。

### 用途2（相性判断）の設計上の考え方

「エスカレーション頻度」「コミュニケーション頻度の好み」「価値判断のパターン」は、**単一のIEDIレコードには存在しない**。これらは複数の取引記録を横断して統計的に浮かび上がる特性である。

したがって、相性判断のためにIEDIレコードにフィールドを追加する必要はない。用途2の実装体は `trust_claims` ツールである——複数のIEDIレコードを集計・分析して「このActorとどう相性が合うか」を AIが定性的に評価する。1件のレコードはデータポイントに過ぎず、相性は多数のデータポイントから導出されるパターンである。

**結論：IEDIは3用途すべてに対して追加フィールドなしで適切。用途2の情報は `trust_claims` が多レコード集計から導出する。**

## Q2：内外統一利用のためのフィールド設計変更案

### 現行設計と内部利用の対応関係

現行設計は外部取引（Requester ↔ Provider の2者）を前提とするが、内部利用（自分が両側）は以下の慣例で同一スキーマに収まる。

**内部利用の慣例：`provider_actor_id = requester_actor_id`**

自分自身が Provider である場合、`provider_actor_id` に自身の DID を設定する。`tool_called` が内部作業の種別を示す（例：`coding_session`, `document_review`, `decision_log`）。これにより、新しいフィールドを追加せず既存の構造で内部・外部を統一的に扱える。

- `provider_prev_record_hash` には自分のチェーン上の直前レコードのハッシュを設定
- 双方向 `record_close` は「1人が両側を計算してハッシュ照合する」として成立する
- Insight の `provider` フィールドは「AIが提案した次回改善点」として使用する

### 追加すべきフィールド

#### 1. `work_domain`（新規フィールド）
```json
"work_domain": {
  "type": "string",
  "enum": ["external_transaction", "internal_task", "decision", "retrospective"]
}
```
- `external_transaction`: 現行の想定通り（Requester ↔ Provider の双方向、異なるActor）
- `internal_task`: コーディング・バックオフィス（`provider_actor_id = requester_actor_id`）
- `decision`: 方針決定・判断記録（`delegated` モード固定）
- `retrospective`: 定期的な振り返り（週次レポート等）

#### 2. Delta フィールドの型変更：オブジェクト → 自然言語文字列

現行 v0.1 の Delta は `tool_called` 固有の構造化オブジェクト（例：`{delivery_time_hours: 23, additional_roundtrips: 1}`）として定義されているが、これをプロトコルレベルで定義すると「スキーマがモデルとドメイン知識に依存する」という問題が生じる。

**変更案：Delta を自然言語の単一文字列フィールドにする。**

```json
"delta": {
  "type": "string",
  "description": "IntentとEvidenceの差分を自然言語で記述する。構造・粒度はモデルが判断する。"
}
```

**利点：**
- Delta の生成・解釈はモデルに委ねる（プロトコルはフォーマットを強制しない）
- `work_domain` や `tool_called` が何であれ同一フィールドで受け取れる
- セマンティック類似度検索（用途3の自動昇格）は文字列埋め込みで対応可能
- プロトコル仕様からドメイン固有知識が消える

**影響範囲：** ホワイトペーパー付録A（JSONスキーマ）の `delta` フィールド定義の変更が必要。`trust_claims` が Delta テキストを LLM で解析して定性評価を生成する形になる。

### 変更しないもの

- コア4フィールド（Intent-Evidence-Delta-Insight）のトップレベル構造
- 3つの実行モード（autonomous / cooperative / delegated）
- 双方向ハッシュ検証の仕組み

内部作業においても `autonomous`（完全自動）から `delegated`（完全手動・人間主導）まで同じモードが適用される。内部作業専用の第4モードは不要。

> **注意（実装スコープとの関係）：** `work_domain` の追加と `delta` の型変更は **ホワイトペーパー v0.2 / IEDIスキーマ v0.2** の変更として定義する。**Approach A（推奨実装）は v0.1 スキーマをベースに `work_domain` と自然言語 Delta を追加した v0.2 ドラフトで実装する。** ホワイトペーパーの付録Aスキーマ更新は実装後に v0.2 として発行する。

## Q3：自動化・高品質化への適用分析

用途3は「記録 → 定期レビュー → スキル反映 → モード判定」という2段階の学習ループで実現する。

---

### Level 1：オフライン学習ループ（定期実行）

蓄積されたIEDIレコードを定期的にレビューし、サービス実行の精度を高める。

```
【定期バッチ処理】

蓄積されたIEDIレコード群
        ↓
  tool_called 単位でグループ化
        ↓
  各レコードの intent / delta / insight / status をレビュー
        ↓
  パターン抽出：
    ・このツールはどういう意図宣言で呼ばれることが多いか
    ・失敗したケースに共通する意図の特徴は何か
    ・Insightに繰り返し現れる改善提案は何か
        ↓
  スキル情報（システムプロンプト / コンテキスト）へ反映
        ↓
  ★ 現行スキルでの成功率を再算出・保存
    （スキル更新後の期待成功率をキャッシュする）
        ↓
  次回実行の品質が向上
```

「スキル情報」とは、`intent_statement_[tool_name]` Promptテンプレートやシステムプロンプトに追記される、過去の知見の集積である。定期レビューによりこれが更新されることで、Intent宣言の質が上がり、結果的に Delta が小さくなる（乖離が減る）という正のフィードバックループが生まれる。

**なぜスキル更新後に成功率の再算出が必要か：**  
スキル更新前に積まれたIEDIレコードの `status` は旧スキルでの実行結果を反映している。オンライン判定（Level 2）がこれをそのまま参照すると、改善されたスキルの実力を過小評価してしまう。スキル更新後に「現行スキルならこのintentに対して成功するか」を再評価してキャッシュすることで、オンライン判定が常に現行スキルの実力を参照できる。

**なぜ再算出結果を IEDIレコード内に保持できないか：**  
IEDIレコードは `record_close` 完了時点で `record_hash`（全フィールドのSHA-256）が確定し、以降の変更は改ざんとして検知される。再算出結果を後からレコードに追記することはハッシュチェーンを破壊するため、IEDI レコードへの書き込みは不可。

**設計方針：`skill_performance_cache`（別ストア）**  
再算出結果は IEDIレコードとは独立した読み取りモデルとして保持する。これは「イミュータブルな一次ログ（IEDI）+ 導出された読み取りモデル（キャッシュ）」というパターンであり、キャッシュが消失しても Level 1 バッチを再実行すれば IEDI レコード + 現行スキルから再構築できる。

```json
// skill_performance_cache（IEDI レコードとは別テーブル / 別ファイル）
{
  "cache_id":              "ULID",
  "actor_id":              "DID（またはローカル識別子）",
  "tool_called":           "string",
  "skill_version_hash":    "現行スキル内容の SHA-256",
  "intent_cluster_id":     "string（埋め込み重心 or ラベル）",
  "estimated_success_rate": 0.0,    // 0.0〜1.0
  "sample_size":           0,       // 算出に使ったサンプル数
  "method":                "dry_run | recent_records | manual",
  "computed_at":           "ISO 8601"
}
```

キーは `(tool_called, skill_version_hash, intent_cluster_id)` の組み合わせ。スキルが更新されるたびに `skill_version_hash` が変わり、新エントリが作成される（旧エントリは参照されなくなるが削除しない）。

**再算出の方法（実装選択肢）：**  
- **recent_records**：スキル更新後に実行された最初の N 件の status から暫定成功率を算出（初期はキャッシュなし → フォールバック後、蓄積次第で更新）  
- **dry_run**：代表的な intent サンプルを現行スキルで評価（実行はしない）して成功予測スコアを付ける  
- Approach A（内部ロガー）の初期実装では `recent_records` で十分。`dry_run` は Approach B 以降で検討する。

---

### Level 2：オンライン判定（estimation時 = `record_start` 処理内）

`record_start` が呼ばれた瞬間に、過去のIEDIレコードを参照してモードを決定する。

**入力：**
- `tool_called`：何のサービスを呼ぶか（フィルタキー）
- `intent`：intent templateから生成された意図の詳細テキスト

**処理フロー：**

```
record_start({tool_called, intent, ...}) 呼び出し
        │
        ▼
① intent テキストの埋め込みベクトルを生成

        ▼
② 過去IEDIレコードから
   ・tool_called が一致するレコードを絞り込み
   ・intent の埋め込みベクトルとのコサイン類似度を計算
   ・上位K件を取得（類似タスクの実績群）

        ▼
③ 成功率を参照
   skill_performance_cache から
   (tool_called, current_skill_version_hash, intent_cluster_id) で検索。
   ・キャッシュあり → cached estimated_success_rate を使用
   ・キャッシュなし（初回 or スキル更新直後）→
       直近 K 件の IEDI status から暫定 success_rate を算出
       （旧スキルの実績であることを踏まえ、保守的な閾値で判定）

        ▼
④ モード判定

   ┌─────────────────────────────────────────────┐
   │ tool_config: delegated_required = true       │
   │   → delegated（常時固定）                    │
   ├─────────────────────────────────────────────┤
   │ work_domain = "decision"                     │
   │   → autonomous 昇格禁止（cooperative が上限）│
   │   → similarity / success_rate に関わらず     │
   │      cooperative または delegated に固定      │
   ├─────────────────────────────────────────────┤
   │ similarity < θ_sim（類似実績なし）           │
   │   → cooperative（未経験領域）                │
   ├─────────────────────────────────────────────┤
   │ similarity ≥ θ_sim かつ                     │
   │ success_rate < θ_succ（経験あり・成功率低）  │
   │   → cooperative（繰り返し失敗パターン）      │
   ├─────────────────────────────────────────────┤
   │ similarity ≥ θ_sim かつ                     │
   │ success_rate ≥ θ_succ                       │
   │   → autonomous（安定した実績あり）           │
   └─────────────────────────────────────────────┘

        ▼
  mode_used を IEDIレコードに記録して返却
```

**閾値 θ_sim / θ_succ の扱い：**  
数値閾値は実装パラメータであり、プロトコル仕様では定義しない。実際のIEDIデータが蓄積される中で経験的にチューニングする（Level 1 のスキル更新プロセスに含めることもできる）。

### 用途3でのIEDI各フィールドの役割

| フィールド | オンライン判定（Level 2）での役割 | オフライン学習（Level 1）での役割 |
|---|---|---|
| **Intent** | 新規タスクとの類似度計算の検索キー | 「このツールが使われる文脈」のパターン抽出元 |
| **Evidence** | （判定には直接使わない） | 実行ログからスキル改善ヒントを抽出 |
| **Delta** | （判定には直接使わない） | Delta テキストの頻出パターン → スキル改善のシグナル |
| **status** | success_rate 計算の基礎データ | 失敗パターンの特定 |
| **Insight** | （判定には直接使わない） | 繰り返し現れる改善提案 → スキル情報へ反映 |

**設計上の注意：** success_rate は `status` フィールド（完了/未完了）から計算する。Delta テキストは Level 1（スキル改善）で使用し、Level 2（モード判定）では直接使用しない。Delta の「良し悪し」の解釈はモデルに委ね、プロトコルは status のみを判定シグナルとして扱う。これにより判定ロジックが Delta の自然言語品質に依存しない。

---

### 用途3特有の制約

方針決定・価値観判断に関わる作業（`work_domain: decision`）は **`cooperative` を上限モード**とする。success_rate・similarity がどれだけ高くても `autonomous` への昇格は行わない。`cooperative`（AIが準備・人間が判断）または `delegated`（人間が主導・AIが補助）の範囲内でモードが決まる。

「この判断を完全に自動化してよいか」はDelta統計では判断できない。最低限、人間が確認・承認するステップ（cooperative以上）を維持する。

## Approaches Considered

> **Effortスケール：** S = 1週間以内 / M = 1〜4週間 / L = 1〜3ヶ月 / XL = 3ヶ月超

### Approach A: 内部IEDIロガー（Solo）★推奨
**概要：** 自分のClaude Codeセッションを手元でIEDIレコードとして記録するCLIツール。`provider_actor_id = requester_actor_id` の慣例を使い、v0.2 スキーマ（`work_domain` + 自然言語 Delta）で実装する。  
**Effort:** S（スキーマ + SQLite + CLI）  
**Risk:** Low  
**Pros:**
- カウンターパーティ不要。今日から使える。
- 用途3（自動化学習）がすぐに得られる
- Delta自然言語化・`work_domain` 追加を実際のデータで検証できる
- v0.2スキーマのドラフト検証として機能する

**Cons:**
- 双方向信用検証のデモができない
- ネットワーク効果・用途1の価値は後フェーズまで見えない

### Approach B: 双方向AMCPサーバー（1ツール）
**概要：** 完全準拠のAMCPサーバーを1つのActor固有ツール（`code_review`等）で実装。2つのAMCノード間で実際に取引できる状態を作る。  
**Effort:** M（サーバー + スキーマ + IEDIストレージ）  
**Risk:** Medium  
**Pros:**
- 5フェーズAMCPのエンドツーエンドを証明
- `record_close`でハッシュ一致を実際に確認できる
- DID + ハッシュチェーンの実装が先行技術として機能する

**Cons:**
- 2ノード必要（最低1つはテストクライアントとして自前で書く）
- 内部作業記録（用途3）はスコープ外

### Approach C: Trust Claims APIのみ（読み取り側優先）
**概要：** `actor_info` / `capabilities` / `trust_claims` / `record_verify` のみ実装。書き込みパス（record_start/feedback/close）は後回し。  
**Effort:** M  
**Risk:** Low  
**Pros:**
- 「信用情報の見せ方」を早期に検証できる
- IEDIを手動投入してUI価値を先に確認

**Cons:**
- 書き込みパスなしではIEDI記録が育たない
- 結局 Approach B も実装が必要になる

---

## Recommended Approach

**Approach A：内部IEDIロガー**

理由：最初の実装は「IEDIの設計が現実の作業記録として機能するか」を検証することが目的。それには実際の作業を記録するのが最短で、カウンターパーティを必要としない。さらに、Delta自然言語化と `work_domain` 追加というスキーマ変更（v0.2）を実データで検証する機会になる。先行技術の「動く証明」として双方向AMCPは次フェーズで十分。

**スコープに含む（Scope In）：**
- IEDIスキーマ v0.2 ドラフト（`work_domain` + `delta: string`）の JSON Schema 定義
- CLIツール: `iedi start --intent "..."` → `record_id` 発行
- CLIツール: `iedi close --delta "..." --insight "..."` → レコードを SQLite に保存
- CLIツール: `iedi query` → 過去レコード一覧・検索
- ハッシュチェーン（`prev_record_hash`）— 単一アクターの自己チェーン
- `provider_actor_id = requester_actor_id` 慣例の実装（内部利用）
- `work_domain` の4値を使ったレコード分類

**スコープ外（Scope Out）：**
- AMCPサーバー・クライアント（ネットワーク通信）
- 双方向ハッシュ検証（record_close の hash_match）

## Open Questions

1. 実装言語の選択：TypeScript（MCP SDK成熟度高、将来のAMCPサーバーと統一できる）を推奨。Python も可。
2. Delta自然言語の粒度・書式：「1文か複数文か」「箇条書きか文章か」は仕様で固定せず、実際の使用から帰納する。最初の数十件から定形パターンが見えてくる。
3. `cancelled` ステータスのコスト負担ルール：内部利用ロガーには関係しない。Approach B（双方向AMCPサーバー）実装時に定義する。
4. **trust_claims の導出モデル（Approach B への接続）：** 内部ロガーで蓄積した IEDIレコードから、`trust_claims` ツールは「`tool_called` でフィルタした過去レコードの Delta テキスト群を LLM が要約 → カテゴリ別の信頼度コメントを生成」という形になる。Delta が自然言語であることで、この集計は LLM が直接テキストを読んで行える。Approach B 完了後に実装。

## Success Criteria

以下がすべて確認できた時点で「最小くさびの実装完了」とする。

1. `iedi start --intent "テストを書く"` → `record_id` が発行される
2. `iedi close --delta "テスト範囲が想定より広く、統合テストを追加した" --insight "次回は事前にインターフェースを確認する"` → SQLite にレコードが保存される
3. 保存されたレコードに `work_domain`、`intent`、`delta`（文字列）、`insight` がすべて含まれている
4. `iedi query` → 過去レコードが一覧表示される
5. 2件以上保存した後、レコードの `prev_record_hash` が前レコードの `record_hash` と一致する（ハッシュチェーンが繋がっている）
6. `provider_actor_id = requester_actor_id` のレコードが `work_domain: internal_task` で正常に保存される

## What I noticed about how you think

- **ホワイトペーパーが問いの形を決めている。** Q1・Q2・Q3を自分で設計して持ち込んだことは、「何が分かっていないかを正確に知っている」ことの証明。これはドメイン知識がないとできない。
- **「先行技術として確立する」という表現を使っている。** 仕様文書の公開日をもってprior artとする設計は、特許戦略を意識した動き。技術仕様と法的保護を同時に考えているのが読み取れる。
- **「AMCはAMCPを話す法人」という表現を使っている。** 技術プロトコルと経済単位を「同一概念の異なる抽象層」と名付けたのはフレーミングの精度が高い。これは「まず作って後から意味を見出す」人の言葉ではなく、設計から思想が先にある人の言葉。
- **IEDIの「Delta はスコアではない」主張を、A2A・ERC-8004と比較して先に封じている。** 競合プロトコルへの反論を仕様書内で展開しているのは珍しい。防衛的でなく、先手を打つ姿勢。
